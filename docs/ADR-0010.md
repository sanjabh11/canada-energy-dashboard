# ADR-0010: PV and Dispatch Closeout Contracts

Status: Accepted
Date: 2026-04-24

## Context

The dispatch and PV closeout branches now depend on deterministic training artifacts,
strict runtime gates, browser smoke coverage, and a shared artifact ledger.
The remaining failure mode is contract drift between Python training, TypeScript
inference, Playwright runtime scope, and the fixture/ledger outputs.

During the closeout we hit a few concrete parity issues:

- The PV fixture rows needed to be regenerated from raw node fields so Python and
  TypeScript both consume the same feature builder.
- `topSuspects` was risk-ranked early, but `topEdges` still sorted alphabetically
  until the final closeout pass.
- The browser trained-flag workaround must be scoped to Playwright's web server
  process, not committed into a normal development env file.
- `label_margin` is a top-two class probability gap from the Python exporter, not
  a threshold-distance metric.
- The Python exporter and TypeScript runtime do not round at identical points, so
  a small raw confidence delta is expected even when ordering and classes match.
- The dispatch artifact SHA had to be computed after `target_mean` and
  `target_std` were embedded, otherwise the published JSON and the canonical SHA
  diverged.
- A local GitHub remote URL exposed a credentialed token during inspection; that
  is a repository hygiene issue and must not be propagated into docs, commits, or
  workflows.

## Decision

We standardize the closeout contract on these rules:

- PV suspect ordering is rank-then-tie-break on 3-decimal risk buckets, with
  alphabetical fallback on node ID.
- PV edge ordering follows the same 3-decimal bucket comparator, with alphabetical
  fallback on `from` / `to`.
- `label_margin` is the top-two class probability gap from the Python exporter.
  The strict `faultClass` assertion only applies when `label_margin >= 0.05`.
- The PV gate floor is intentionally lower than the current artifact so the
  simulator-calibrated model remains accepted, but it still blocks materially
  weaker retrains. The current floor is `f1 >= 0.32`, `top3 >= 0.52`, and
  `validation_loss <= 0.09`.
- The PV fixture may tolerate a small raw confidence delta because the exporter
  and runtime quantize at different points, but exact ordering must still match.
  The current test tolerance is `<= 5e-3`, with a looser historical note retained
  only as session context.
- The dispatch and PV soak gates read fallback behavior from `job_execution_log`,
  and production flag flips remain blocked until the 7-day rolling fallback rate
  is below 1%.
- The shared artifact ledger is written through `scripts/record-artifact.mjs` so
  both training workflows publish the same row shape keyed by
  `model_key, model_version, artifact_sha256`.
- Trained browser flags belong in `playwright.config.ts` `webServer.env`; they
  do not belong in a committed `.env.development` file.
- Remote `workflow_dispatch` verification is part of the closeout definition:
  both training workflows must be run against real repo secrets, and the
  resulting `ml_model_artifacts` rows must be confirmed before the branch is
  considered finished.

## Consequences

- Python and TypeScript must stay in lockstep on both suspect and edge ordering.
- The PV conformance fixture is a contract artifact, not just a snapshot; it must
  be regenerated whenever the exporter or ranking comparator changes.
- Playwright component tests stay deterministic without leaking trained flags into
  everyday local dev.
- Soak workflows become the source of truth for trained-path health instead of
  ad hoc local checks.
- The artifact ledger can be updated consistently from both training workflows
  without inline duplication.

## Follow-up

- Keep the `/resilience` PV contract chip and smoke test in sync with the
  regenerated fixture.
- Keep the closeout scoreboard in `docs/scoreboard.md`; `docs/codex_plan.md` is
  a working memo, not the authoritative scorecard.
- If workflow-dispatch artifact insertion ever fails in CI, treat the repo secret
  configuration as the first suspect before changing the training scripts.
